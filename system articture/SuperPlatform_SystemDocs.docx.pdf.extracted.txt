1. System Overview
This platform is a comprehensive Multi-Role Super App combining services, digital wallet, job
marketplace, agent network, and multi-country administrative control into a single unified system.
1.1 Platform Identity
Attribute 	Description
Type 	Super App + Fintech + Marketplace Hybrid
Architecture 	Modular Microservices — Independently deployable modules
Design Principle 	Config-driven — No hardcoded business logic
Scale 	Multi-country, Multi-currency, Multi-language ready
Access Control 	Role-Based Access Control (RBAC) — 8 distinct roles
Financial Safety 	Transaction-safe — Atomic operations, Idempotency keys
1.2 Key Design Principles
• Modular system — each module independently deployable and scalable
• Config-driven — all business rules stored in DB, no hardcoding
• Multi-country ready — services toggled per country/region
• Role-based architecture — 8 roles with granular permission control
• Transaction-safe — atomic wallet operations with rollback support
• Event-driven commission — async processing via message queues

4. Core Modules — Developer Reference
Each module is an independent microservice. Modules communicate via REST APIs internally and
event queues for async operations.
Module 1: Auth Service
Central authentication and authorization gateway for all services.
API Endpoints POST /auth/login | POST /auth/register | POST /auth/otp/verify | POST
/auth/refresh | POST /auth/logout
Dev Notes All other services validate JWT from this service. Role permissions cached in
Redis. 2FA/OTP via SMS gateway.
Module 2: User & Member Service
Manages user profiles, membership upgrade, and KYC processing.
API Endpoints GET /users/:id | PUT /users/:id | POST /users/kyc/submit | GET
/users/kyc/status | POST /users/upgrade-member
Dev Notes KYC document stored in object storage (S3/GCS). On KYC approval, emits
MEMBER_APPROVED event consumed by Wallet Service.
Module 3: Wallet Service
Handles digital wallet creation, balance management, and fund transfers.
API Endpoints POST /wallet/add-money | POST /wallet/transfer | POST /wallet/withdraw
| GET /wallet/balance | GET /wallet/history
Dev Notes Wallet only created for MEMBER+ roles. Uses optimistic locking (version
column) on balance updates. All transfers are atomic.
Module 4: Service Engine
Config-driven utility services: mobile recharge, DTH, bill payments.
API Endpoints GET /services | POST /services/recharge | POST /services/bill-pay |
GET /services/history | GET /services/providers
Dev Notes
Services list fetched from service_config JOIN country_config. Integrates with
BBPS and operator APIs. Commission triggered on each successful service
transaction.
Module 5: Task Service
Core task booking and Saathi assignment engine.

API Endpoints POST /tasks | GET /tasks/:id | PATCH /tasks/:id/status | GET

Business marketplace for job postings and advertisement campaigns.
API Endpoints POST /jobs | GET /jobs | POST /jobs/:id/apply | POST /ads/campaign |
GET /ads/analytics
Dev Notes All posts require Admin approval before going live. Ad billing deducted from
Business wallet in real-time. Targeting by country, state, district, and role type.
Module 8: Partner Service
Multi-level partner hierarchy management and monitoring.
API Endpoints POST /partners/assign | GET /partners/hierarchy | GET /partners/analytics
| GET /partners/earnings
Dev Notes Hierarchy: Admin → Country Head → State Partner → District Partner. Each
level only sees downward data. Commission aggregated up the hierarchy.
Module 9: Country Config Service
Country-specific service visibility and configuration management.
API Endpoints POST /country | PUT /country/:id/services | GET /country/:id/config |
GET /services/available
Dev Notes All service visibility queries JOIN this table. Stores currency, KYC rules, enabled
services as JSON config per country.
Module 10: Admin Service
Full system control, analytics, and administrative operations.

7. Critical Business Logic Rules
These rules MUST be enforced in code. They are non-negotiable business requirements.
7.1 Wallet Access Rules
Rule 	Valid 	Invalid
Wallet creation 	Only after KYC APPROVED 	Cannot create for USER role
Wallet access 	MEMBER role and above 	USER role — 403 Forbidden
Concurrent transactions Optimistic lock (version
column)
Direct UPDATE without version
check
Duplicate payment Idempotency key blocks
duplicate Processing same ref_id twice
7.2 Commission Distribution Rules
• Commission is calculated as a percentage of the transaction/task amount
• Percentages are fetched from commission_rules table — never hardcoded
• Commission is distributed asynchronously via message queue (not in the transaction flow)
• If a hierarchy level has no assigned partner, that commission goes to Admin
• Commission ledger is append-only — entries are NEVER updated or deleted
• Commission is only paid on COMPLETED + CONFIRMED transactions/tasks
7.3 Service Visibility Rules
• Services are only shown to users if service_config.enabled = TRUE for their country
• Service list API MUST JOIN with country_config on every request — no caching of service lists
without TTL
• Country Head can enable/disable services at any time — changes take effect immediately
• If a user changes country, their service list is refreshed on next app load
7.4 Task Assignment Rules
• Task amount is held in escrow (separate wallet hold) the moment the task is created
• Escrow is released to Saathi wallet only after User confirms completion (status = CONFIRMED)
• If User does not confirm within 48 hours after COMPLETED, auto-confirm triggers
• If no Saathi accepts within 15 minutes: expand search radius from 2km to 5km, notify again
• If no Saathi accepts within 30 minutes: task moved to OPEN pool, notify all Saathi in district
• Task cancellation by User before acceptance: full refund, no commission
• Task cancellation by User after Saathi acceptance: partial charge (10%) to Saathi as
compensation

• Synchronous: REST/HTTP between services for real-time requests (auth validation, user lookup)
• Asynchronous: Message queue for commission events, notifications, audit logs
• Service discovery: Kubernetes DNS or API Gateway routing
• Each service has its own database — no cross-service DB joins
• Shared data accessed via service-to-service API calls only
