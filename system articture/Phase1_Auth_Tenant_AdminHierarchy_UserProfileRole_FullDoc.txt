3) MICRO-FEATURES (DETAILED)
==================================================

-----------------------------
M1: Auth & Session Core
-----------------------------
F1.1 Send OTP endpoint
- Validate mobile format
- Check resend cooldown
- Generate OTP, hash OTP, store temporary state
- Trigger SMS service

F1.2 Verify OTP endpoint
- Validate OTP payload
- Verify hash against stored OTP
- Enforce max attempts
- Mark mobile verification window active

F1.3 User registration endpoint
- Accept required profile fields
- Ensure mobile is OTP verified
- Check duplicate mobile
- Create user with default identity USER
- Attach default functional role USER
- Return JWT + normalized user summary

F1.4 User login endpoint
- Tenant-aware credential lookup
- Verify password hash
- Return JWT payload with identity + roles + tenant context

F1.5 Logout endpoint
- Stateless acknowledgement for JWT flow
- Optional blacklist hook placeholder

F1.6 Password reset flow
- forgot-password: OTP re-issue for existing user
- reset-password: strong password validation + update hash

Acceptance Criteria (M1)
- All endpoints return consistent success/error shape
- JWT contains user_id, identity, roles, tenant_id, parent_id
- Registration/login both tenant-aware and audit logged


-----------------------------
M2: OTP & Credential Security
-----------------------------
F2.1 OTP storage in Redis
- Store hashed OTP
- TTL expiry
- Attempt counter

F2.2 OTP rate limiter
- Per mobile + per IP 30 second cooldown
- Block repeated spam requests

F2.3 Password policy
- Min length 8+
- Uppercase required
- Number required
- Special char required

F2.4 Bcrypt hashing policy
- Use secure work factor
- No plaintext leakage in logs

Acceptance Criteria (M2)
- Expired OTP rejected
- Max attempts lock works
- Weak passwords rejected with human-readable message


-----------------------------
M3: Tenant Resolution & Isolation
-----------------------------
F3.1 Host/domain extraction middleware
- Extract domain from request host
- Ignore port component

F3.2 Tenant lookup
- Resolve tenant by domain
- Reject unknown domain

F3.3 Request context propagation
- Attach tenant_id in request lifecycle

F3.4 Tenant-safe queries
- Login and protected lookups filter by tenant_id

Acceptance Criteria (M3)
- Invalid domain requests blocked
- Cross-tenant authentication disallowed
- Audit logs carry tenant_id where applicable


-----------------------------
M4: Hierarchical Identity Management
-----------------------------
F4.1 ADMIN creates STATE_PARTNER
F4.2 ADMIN/STATE_PARTNER create DISTRICT_PARTNER
F4.3 ADMIN/STATE_PARTNER/DISTRICT_PARTNER create AGENT
F4.4 ADMIN/STATE_PARTNER/DISTRICT_PARTNER/AGENT create USER

F4.5 Parent linkage logic
- If parentId provided, validate same tenant
- If not provided, default parent to creator

F4.6 Hierarchy metadata
- Store createdBy and parentId

Acceptance Criteria (M4)
- Unauthorized identity creator blocked with 403
- Parent validation enforces tenant boundary
- Created user returns identity summary


-----------------------------
M5: User Profile & Functional Role Management
-----------------------------
F5.1 Get my profile
- Return safe profile (no password)
- Include roles relation

F5.2 Assign functional role
- Protected endpoint for authorized operator
- Upsert role mapping

F5.3 Role list in response
- Return roles as readable names

F5.4 Future-safe role expansion
- Keep identity and functional roles separate

Acceptance Criteria (M5)
- Profile endpoint always tenant-safe
- Role assignment idempotent for same role


-----------------------------
M6: Authorization Layer (Identity + Role)
-----------------------------
F6.1 JWT verification middleware
- Parse bearer token
- Decode/verify signature
- Attach req.user

F6.2 Identity guard
- Validate req.user.identity against allowed identities

F6.3 Role guard
- Validate req.user.role OR roles[] for functional permission

F6.4 Authorization consistency patch
- Decide one policy for privileged user routes:
  Option A: Identity-first guard for hierarchy actions
  Option B: Role-first guard for business permissions

Recommended for Phase 1:
- Identity guard for hierarchy/admin creation routes
- Role guard for functional role assignment and business-style permissions

Acceptance Criteria (M6)
- No mixed ambiguous guard behavior on same action type
- 401 vs 403 responses are consistent


-----------------------------
M7: Audit Logging & Observability
-----------------------------
F7.1 Log security-sensitive actions
- USER_REGISTER
- USER_LOGIN
- CREATE_STATE_PARTNER
- CREATE_DISTRICT_PARTNER
- CREATE_AGENT
- CREATE_USER
- ROLE_ASSIGNMENT

F7.2 Structured metadata
- Include actor, target, tenant, lightweight metadata

F7.3 Error logging hygiene
- Avoid leaking secrets/passwords/OTP

Acceptance Criteria (M7)
- Sensitive actions are visible in audit table
- Failed audit insert does not crash primary flow


-----------------------------
M8: API Standardization & Error Contract
-----------------------------
F8.1 Unified success response shape
- success: true
- data OR endpoint-specific payload

F8.2 Unified error response shape
- success: false
- message

F8.3 HTTP status semantics
- 200/201 success
- 400 validation
- 401 unauthenticated
- 403 unauthorized
- 404 not found
- 409 conflict
- 500 internal error

F8.4 Input validation baseline
- Mobile
- Required fields
- Identity enum
- Date parsing guard

Acceptance Criteria (M8)
- Same contract across all phase-1 endpoints
- Frontend can parse errors uniformly
